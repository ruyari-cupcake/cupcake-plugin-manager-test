## 1. 설정 UI 표시 문제 — 발견 사항

### ✅ 이미 수정된 것 (v1.22.28에 반영)

**문제:** 서브플러그인 설정 화면이 아예 안 뜨는 문제  
**원인:** 서브플러그인이 "직원증 번호"와 "이름" 두 가지 신분을 가지고 있는데, 설정 화면이 직원증 번호로만 찾으려 해서 서로 못 만났음  
**비유:** 마치 회사에서 본명은 "김철수"인데 사원번호가 "EMP_20250315_abc"이고, 회의실 문에는 사원번호로 이름표를 달았는데, 정작 본인이 "나 김철수야" 하고 찾으러 와서 못 들어간 것  
**수정:** 사원번호로 못 찾으면 이름으로도 찾아보게 변경 → 성공

### ❌ 아직 남은 문제 1개

**Navigation 플러그인 이름에 이모지 prefix**  
- Navigation 플러그인이 자기 이름을 `🧭 Chat Navigation`으로 등록함
- 시스템에 저장된 이름은 `CPM Component - Chat Navigation`
- "🧭 Chat Navigation"과 "CPM Component - Chat Navigation"은 서로 포함 관계가 아니라서 이름 매칭도 실패
- **비유:** "나 🧭 항해사야" vs "CPM 부서 - 항해사님" → 서로 다른 사람으로 인식
- **영향:** Navigation 플러그인의 설정 UI가 안 나올 수 있음 (단, Navigation은 원래 설정 UI가 별로 없어서 실질적 영향은 작음)
- **해결법:** Navigation 이름에서 이모지를 빼면 됨

### 서브플러그인 ID·이름 매칭 현황

## 2. 백그라운드에서 돌아가는 것들

> "이 앱이 백그라운드에서 뭘 하고 있나?" → 아래 표 확인

### 상시 동작하는 감시 프로그램 (MutationObserver)

**MutationObserver란?** 화면(DOM)에 뭔가 바뀔 때마다 자동으로 알려주는 "감시카메라" 같은 것. 채팅이 추가되거나 버튼이 생기면 감지함.

| 감시자 | 뭘 감시하나 | 언제 꺼지나 | 위험도 |
|--------|-----------|-----------|--------|
| **Chat Limiter** | 채팅 메시지 추가/삭제 | 플러그인 꺼질 때 ✅ 정상 해제 | 🟢 안전 |
| **Chat Resizer** | 텍스트 입력창 생성 | 플러그인 꺼질 때 ✅ 정상 해제 | 🟢 안전 |
| **Chat Navigation** | 채팅 화면 변화 | 플러그인 꺼질 때 ✅ 정상 해제 | 🟢 안전 |

**결론:** 감시카메라 3대 = 적정 수준. 각각 제 역할이 있고, 정리도 잘 됨.

### 반복 타이머 (setInterval / setTimeout)

| 타이머 | 용도 | 주기 | 꺼지나? | 위험도 |
|--------|------|------|---------|--------|
| **Navigation 폴링** | 채팅 스크롤 영역 찾기 | 찾을 때까지 반복 | 찾으면 자동 중지 ✅ | 🟢 안전 |
| **API 재시도** | 실패한 API 호출 재시도 | 700ms × 시도횟수 | 3회 실패 시 중단 ✅ | 🟡 보통 |
| **번역 캐시 TTL** | 캐시 유효 시간 | 2분마다 갱신 가능 | 사용 시에만 작동 ✅ | 🟢 안전 |

**결론:** 상시 돌아가는 반복 타이머는 없음. 전부 "필요할 때만" 동작.

## 3. 메모리(RAM) 사용 관련

> "오래 쓰면 렉 걸릴 수 있나?" → 가능성 있는 부분 확인

### 🟢 안전한 것들

| 항목 | 설명 | 예상 메모리 |
|------|------|-----------|
| 메인 플러그인 코드 | 한 번 로드되면 추가 증가 없음 | ~610KB (고정) |
| 프로바이더 등록 | 모델 목록 한 번 등록 | ~50KB (고정) |
| Copilot 토큰 캐시 | 1개만 저장 | ~1KB (고정) |

### 🟡 주의가 필요한 것들

| 항목 | 설명 | 왜 위험한가 | 예상 크기 |
|------|------|-----------|----------|
| **서브플러그인 코드 저장** | 플러그인 코드 문자열을 메모리에 저장 | 핫 리로드(재시작) 시 이전 코드가 안 지워짐 | 플러그인당 ~50KB × 재시작 횟수 |
| **번역 캐시 전체 로드** | 번역 데이터 전부를 한번에 불러옴 | 번역이 1만건이면 ~500KB 한번에 로드 | 사용량에 비례 |
| **Copilot 토큰 상태 캐시** | 토큰별 상태 저장 (Map) | 토큰 많이 넣으면 Map이 커짐 (정리 없음) | 토큰당 ~1KB |

**비유:**  
- "서브플러그인 코드 저장" = 새 버전 깔 때 이전 버전을 휴지통에 안 버리고 쌓아두는 것
- "번역 캐시" = 도서관에서 책 한 권만 필요한데 도서관 전체 장서 목록을 가져오는 것
- **일반 사용에서는 문제 없음**, 장시간(수 시간) + 잦은 새로고침 시에만 영향

## 4. 네트워크 트래픽

> "인터넷 데이터를 많이 먹나?" → 일반적으로는 안 먹음

### 정상 동작 시 네트워크 호출

| 시점 | 무엇을 | 얼마나 |
|------|--------|--------|
| 플러그인 시작 시 | 업데이트 번들 확인 | 1회 (~960KB) |
| 사용자가 수동 클릭 시 | 업데이트 확인 | 1회씩 |
| AI 대화 시 | 프로바이더 API 호출 | 대화당 1회 |
| Copilot 토큰 교환 | GitHub API | 토큰당 1회 |

### ⚠️ 주의 상황: API 키 여러 개 + 실패

| 키 개수 | 실패 시 재시도 | 최대 API 호출 수 |
|---------|--------------|-----------------|
| 1개 | 3회 | 3회 |
| 3개 | 각 키당 3회 | **9회** |
| 5개 | 각 키당 3회 | **15회** |

**설명:** API 키를 여러 개 등록했을 때, 한 키가 실패하면 다음 키로 넘어감. 키가 5개면 최악의 경우 한 번의 대화에 15번 API를 찔러봄.  
**비유:** 열쇠를 5개 갖고 있는데 문이 안 열려서 5개를 각각 3번씩 돌려보는 것  
**일반 사용에서는 문제 없음.** 키가 1-2개면 최대 6회로 합리적.

## 5. 이벤트 리스너 관리 (정리 안 되는 문제)

> "이벤트 리스너란?" → 사용자가 버튼 클릭, 키보드 입력 등을 할 때 반응하는 "대기 직원". 예를 들어 "슬라이더를 움직이면 메시지 개수를 바꿔라"는 명령을 받고 대기.

### 문제: 플러그인 재시작 시 "이전 대기 직원"이 퇴근을 안 함

| 플러그인 | 어떤 리스너 | 정리됨? | 영향 |
|----------|-----------|---------|------|
| **Chat Limiter** | 설정창 체크박스/슬라이더 | ❌ 안 됨 | 재시작마다 중복 반응 가능 |
| **Chat Resizer** | 텍스트 영역 버튼 | ❌ 안 됨 | 버튼 클릭 시 중복 동작 가능 |
| **Chat Navigation** | 키보드, 드래그 | ✅ 정리됨 | 안전 |
| **설정 패널 전체** | 업로드/삭제/토글 | ⚠️ DOM 교체로 암묵적 정리 | 대부분 안전, 빠른 연타 시 위험 |

**비유:**  
- "리스너 정리 안 됨" = 식당에서 웨이터를 새로 배치했는데 이전 웨이터가 안 나감 → 같은 주문을 2번 받아옴
- **실제 영향은?** 일반 사용에서는 거의 체감 불가. 설정창을 열었다 닫았다 반복하거나, 플러그인을 핫 리로드(수동 재시작)하는 경우에만 해당.

### 페이지 종료 시 정리 등록 현황

| 플러그인 | risuai.onUnload 등록 | 의미 |
|----------|---------------------|------|
| Chat Limiter | ✅ 있음 | 페이지 닫을 때 정리함 |
| Chat Navigation | ❌ 없음 | 페이지 닫을 때 정리 안 함 (브라우저가 대신 처리) |
| Chat Resizer | ❌ 없음 | 페이지 닫을 때 정리 안 함 (브라우저가 대신 처리) |

**설명:** `risuai.onUnload`는 "페이지 나갈 때 뒤처리 해줘" 라는 등록. 없어도 브라우저가 페이지를 닫으면 전부 지워지므로, 일반적으로 문제 없음. SPA(페이지 새로고침 없이 화면 전환) 환경에서만 영향.

## 6. Gemini 이중 재시도 문제

**설명:** Gemini 프로바이더에는 `geminiRequest` (외부 함수)와 `geminiFetch` (내부 함수) 두 단계에 각각 재시도 로직이 있음.

```
사용자 → geminiRequest (3회 재시도) → geminiFetch (2가지 경로) → Google API
```

**비유:** 택배를 보내는데, 택배회사가 3번 재배송하고, 택배원도 각각 2번 재시도 → 최대 6번 시도  
**영향:** 일반적으로는 1-2회에 성공하므로 체감 없음. 서버 장애 시에만 불필요한 재시도가 늘어남.

## 7. AWS 대용량 응답 문제

**설명:** AWS 프로바이더는 먼저 응답 전체를 메모리에 받은 다음(collect-first) 처리함. 스트리밍(조금씩 받으면서 처리)이 아님.

**비유:** 유튜브 영상을 스트리밍(바로 재생)하는 대신 전부 다운로드한 후 재생하는 것  
**영향:** 대부분의 AI 응답은 수 KB로 작아서 문제 없음. 초대용량 응답(10MB+)에서만 일시적 메모리 폭증 가능.

## 10. 결론

**CPM v1.22.28은 일반 사용에서 렉, 트래픽 과다, 메모리 폭증 등의 심각한 문제가 없습니다.**

주요 수정 포인트는:
1. Navigation 이모지 이름 → 1줄 수정으로 해결 가능

3. API 키 다수 사용 시 재시도 폭증 가능성 → 키 1-2개면 문제 없음

**긴급하게 수정해야 할 것은 #1(Navigation 이모지)뿐이고, 나머지는 품질 개선 사항입니다.**
