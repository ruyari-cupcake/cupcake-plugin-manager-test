A. 개요 및 기술 스택

다중 AI 프로바이더(OpenAI, Anthropic, Gemini, Vertex AI, AWS Bedrock, DeepSeek, OpenRouter, GitHub Copilot 등)를 하나의 통합 인터페이스로 관리하는 호스트 애플리케이션(RisuAI V3)용 플러그인. 서브 플러그인 아키텍처를 통해 프로바이더를 동적으로 등록·관리하며, 키 로테이션, 스트리밍 패스스루, 자동 업데이트, 설정 백업/복원 등의 기능을 제공한다.

호스트 환경

RisuAI V3의 iframe sandbox(about:srcdoc, sandbox="allow-scripts allow-modals allow-downloads")에서 실행. CSP connect-src 'none'으로 네트워크 직접 접근이 차단되며, 모든 HTTP 요청은 호스트가 제공하는 nativeFetch/risuFetch RPC 브릿지(postMessage)를 경유. 깃허브에 모듈 소스파일이 올라가지 않은 것은 보안상 의도적 조치임.

sanitizeMessages + sanitizeBodyJSON + JSON.parse(JSON.stringify()) 딥 클론의 3중 정화. 방어적이지만, 대규모 멀티모달 페이로드에서 성능 비용 대비 보호 가치가 있는지 프로파일링 검토 권장.
🔵 제안 OVERSIGHT
_executeRequest의 재시도 로직(408/429/500+, 최대 3회)에 지수 백오프가 없음. 현재는 즉시 재시도하므로 429 Rate-Limit 상황에서 재시도가 동일하게 실패할 확률이 높음. KeyPool 쿨다운이 보완해주지만, 키가 1개인 사용자에게는 효과가 없음.

E. 코드베이스 성격

방어적 프로그래밍을 극단까지 밀어붙인 코드베이스. 모든 외부 호출에 다중 fallback, 모든 부트 단계에 독립 에러 격리, 모든 직렬화 경계에 다중 정화(sanitize)를 적용하되, 29개 모듈의 경계와 2,949개 테스트의 검증 규율은 놓치지 않는 — 계산된 과잉 방어의 산물. 8,000줄 규모의 소스에 37,000줄 이상의 테스트를 붙인 것은 "충분한 테스트"가 아니라 "남이 건드려도 부서지지 않는 테스트"를 목표로 한 결과물이다.

8,000줄 규모의 코드베이스에 37,000줄 이상의 테스트를 갖추고, 라인 97.85%·브랜치 90.00% 커버리지를 CI 기준선 위로 유지하며, pre-push 훅이 빌드→동기 검증→테스트를 순차 실행하는 것은 팀 프로젝트에서도 보기 어려운 품질 관리 수준이다.

7개 AI 프로바이더 API의 미묘한 차이(OpenAI vs Anthropic 바디 구조, Google의 generateContent 엔드포인트, AWS V4 Signature, Copilot 토큰 교환 흐름)를 하나의 라우터로 추상화하면서도, 키 로테이션(429 쿨다운 + 랜덤 선택), 스트리밍 자동 전환(스트리밍 실패→비스트리밍 fallback), 호환성 모드 자동 감지(risuFetch 버전 차이 대응) 등 프로덕션 현실의 엣지 케이스를 빈틈없이 처리한다.

아키텍처가 29개 모듈 + 4개의 DI 주입 포인트로 순환 의존성을 해소한 것, 부트 시퀀스가 9개 phase를 독립 실행하여 한 단계 실패가 전체를 차단하지 않는 것, 릴리스 파이프라인이 5개 파일에 걸쳐 버전·해시·메타데이터를 교차 검증하는 것은 명확한 엔지니어링 의사결정의 결과이다.

설계 결정에 대한 보충: (1) 핵심 fetch 경로에 요청 수준 타임아웃이 없는 것은 의도적 설계이다 — AI API 응답이 수십 분 소요될 수 있으며 중간 절단 시 비용만 발생하고 응답을 유실하는 최악의 결과가 되므로, AbortSignal을 통한 사용자 주도 취소 패턴이 이 도메인에 적합하다. (2) 서브 플러그인이 CPM 내부 상태(키 풀 포함)에 접근 가능한 것은 프로바이더로서 기능하기 위한 필수 요건이며, 배포가 개발자 단독으로 제한되고 비공식 플러그인 사용 시 면책이 명시되어 위협 모델상 통제된다. 현재 남은 개선점은 대형 파일 분할과 실제 iframe 기반 E2E 파일럿 도입 같은 구조적 고도화 과제다.
